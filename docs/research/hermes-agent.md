# Hermes Agent (NousResearch/hermes-agent) — primary-source research

Date: 2026-07-21. Sources: repo at commit `d604141` (2026-07-21), https://github.com/NousResearch/hermes-agent, docs read from a clone of `main`; GitHub API for stats.

**Verdict.** Hermes Agent is the strongest dispatcher candidate examined so far: the cron/dispatch loop curia describes already exists inside it twice over. A single long-lived Python gateway process hosts (1) a cron scheduler ticking every 60s with delivery to any chat platform, (2) HMAC-authenticated webhook triggers, and (3) a SQLite kanban board with an embedded dispatcher that claims `ready` tasks each tick and spawns worker agents with retries, heartbeats, PID crash detection, runtime caps, and concurrency limits. Discord is a first-class platform plugin (bot, threads, buttons for HITL questions, voice channels, home channel for cron delivery), and state-between-runs is the project's entire thesis — SQLite FTS5 transcripts, curated MEMORY.md/USER.md, self-created skills, cross-session search. The catch: dispatched workers are **Hermes agents** (`hermes -p <profile> chat -q "work kanban task <id>"`), not Claude Code/Codex processes. Driving external coding agents happens one level down — the agent's terminal tool has an explicit PTY mode "for interactive CLI tools (Codex, Claude Code, Python REPL)", and an optional `codex_app_server` runtime hands whole turns to a Codex subprocess — so curia's Claude Code workers would be agent-mediated, not a first-class provider abstraction like Paseo's. Main caveats: a very large Python monolith (gateway/run.py alone is ~23k lines), ~24k open issues/PRs, no Paseo-style live multi-client attach to one session (multi-device viewing is delegated to Discord itself), and unverified cron catch-up semantics after downtime.

## Role fit

- **Dispatcher: native, three overlapping mechanisms.**
  - **Cron:** `hermes cron create "<cron-expr>" "<prompt>" --deliver discord` — the gateway "calls this [tick()] every 60 seconds from a background thread", with a file lock (`~/.hermes/cron/.tick.lock`) so overlapping processes can't double-fire ([cron/scheduler.py](https://github.com/NousResearch/hermes-agent/blob/main/cron/scheduler.py) module docstring). Jobs persist in `~/.hermes/cron/jobs.json`, outputs in `~/.hermes/cron/output/{job_id}/{timestamp}.md` ([cron/jobs.py](https://github.com/NousResearch/hermes-agent/blob/main/cron/jobs.py)). Jobs support natural-language intervals, per-job skills, Python pre-processing scripts whose stdout becomes agent context, a `[SILENT]` no-notify pattern, and delivery to telegram/discord/slack/sms/email/github_comment/webhooks/local files ([hermes-already-has-routines.md](https://github.com/NousResearch/hermes-agent/blob/main/hermes-already-has-routines.md) — a first-party comparison doc claiming feature parity plus with Claude Code Routines).
  - **Webhooks:** an aiohttp server that validates HMAC signatures (V2 with replay protection), rate-limits, dedupes retries via an idempotency cache, templates payloads into agent prompts, and routes responses to any platform; `deliver_only` routes skip the LLM entirely ([gateway/platforms/webhook.py](https://github.com/NousResearch/hermes-agent/blob/main/gateway/platforms/webhook.py) docstring).
  - **Kanban board + dispatcher — this is curia's ticket-frontier loop, already built.** Tasks live in SQLite (`~/.hermes/kanban.db`, multiple boards under `kanban/boards/<slug>/`) with status (`todo/ready/running/scheduled/blocked/done/archived`), priority, assignee (a Hermes profile), dependencies (`task_links`), comments, events, attempt history, retries, per-task model override, workspace binding (`scratch | worktree | dir:<path>`), and workflow templates ([hermes_cli/kanban_db.py](https://github.com/NousResearch/hermes-agent/blob/main/hermes_cli/kanban_db.py), [hermes_cli/kanban.py](https://github.com/NousResearch/hermes-agent/blob/main/hermes_cli/kanban.py); design spec at [docs/hermes-kanban-v1-spec.pdf](https://github.com/NousResearch/hermes-agent/blob/main/docs/hermes-kanban-v1-spec.pdf)). The gateway embeds the dispatcher: `_kanban_dispatcher_watcher` ticks every `kanban.dispatch_interval_seconds` (default 60), holds a machine-global singleton advisory lock (`.dispatcher.lock`) so only one gateway dispatches, and calls `dispatch_once` per board ([gateway/kanban_watchers.py](https://github.com/NousResearch/hermes-agent/blob/main/gateway/kanban_watchers.py)). Config caps: `max_spawn`, `max_in_progress`, `max_in_progress_per_profile`, `failure_limit` (repeated failures → `blocked` for a human decision), `dispatch_stale_timeout_seconds`, `default_assignee`, and LLM `auto_decompose` with a per-tick cap and a live kill-switch. An external `hermes kanban daemon` replaces the embedded loop when `kanban.dispatch_in_gateway: false` ([docs/kanban/multi-gateway.md](https://github.com/NousResearch/hermes-agent/blob/main/docs/kanban/multi-gateway.md)).
- **Can it drive other agents? Yes, with a caveat.** The default worker spawn is `hermes -p <profile> chat -q "work kanban task <id>"` with `HERMES_KANBAN_TASK/DB/BOARD/WORKSPACE` injected and cwd pinned to the task workspace (`_default_spawn` in [hermes_cli/kanban_db.py](https://github.com/NousResearch/hermes-agent/blob/main/hermes_cli/kanban_db.py)) — workers are Hermes agents, and they report back themselves via kanban tools (`complete`/`block`, [tools/kanban_tools.py](https://github.com/NousResearch/hermes-agent/blob/main/tools/kanban_tools.py)). To reach Claude Code/Codex: (a) the terminal tool's PTY mode is documented as being "for interactive CLI tools (Codex, Claude Code, Python REPL)" on local and SSH backends ([tools/terminal_tool.py](https://github.com/NousResearch/hermes-agent/blob/main/tools/terminal_tool.py) lines 976, 3100), so a Hermes worker can drive a Claude Code session; (b) `/codex-runtime` flips `model.openai_runtime` to `codex_app_server`, handing entire turns to a Codex subprocess ([hermes_cli/codex_runtime_switch.py](https://github.com/NousResearch/hermes-agent/blob/main/hermes_cli/codex_runtime_switch.py)); (c) `delegate_task` spawns parallel isolated Hermes subagents in-process ([tools/delegate_tool.py](https://github.com/NousResearch/hermes-agent/blob/main/tools/delegate_tool.py)); (d) `dispatch_once(spawn_fn=...)` is a code-level seam for a custom worker launcher — but it is a Python API, not config, so pointing the dispatcher directly at `claude -p` means writing code against Hermes internals.
- **Worker host: native** for Hermes-flavored workers, with six terminal backends — local, Docker, SSH, Singularity, Modal, Daytona ([tools/environments/](https://github.com/NousResearch/hermes-agent/blob/main/tools/environments), README feature table).
- **Bridge: native.** The gateway *is* a Discord/Telegram/Slack/WhatsApp/Signal bridge (see below), including a blocking HITL `clarify` tool rendered as inline buttons with timeouts ([tools/clarify_tool.py](https://github.com/NousResearch/hermes-agent/blob/main/tools/clarify_tool.py), [tools/clarify_gateway.py](https://github.com/NousResearch/hermes-agent/blob/main/tools/clarify_gateway.py)).

## State between runs

**This is the product's thesis** ("The self-improving AI agent… builds a deepening model of who you are across sessions", [README.md](https://github.com/NousResearch/hermes-agent/blob/main/README.md)). Layers, all file/SQLite-backed under `~/.hermes` (per-profile `HERMES_HOME`):

- **Transcripts:** SQLite `state.db` in WAL mode with an FTS5 virtual table — session metadata, full message history, model config, per-source tagging (`cli`, `telegram`, `discord`, …) ([hermes_state.py](https://github.com/NousResearch/hermes-agent/blob/main/hermes_state.py) docstring). `sessions.json` maps conversation lanes to session ids ([docs/session-lifecycle.md](https://github.com/NousResearch/hermes-agent/blob/main/docs/session-lifecycle.md) §3).
- **Curated memory:** `MEMORY.md` (environment facts, conventions) + `USER.md` (user model), bounded, injected into the system prompt as a frozen snapshot at session start; mid-session writes are durable immediately ([tools/memory_tool.py](https://github.com/NousResearch/hermes-agent/blob/main/tools/memory_tool.py) docstring). Periodic "nudges" prompt the agent to persist knowledge (README).
- **Procedural memory:** autonomous skill creation after complex tasks, skills that self-improve during use, agentskills.io-compatible (`skills/`, `optional-skills/`, [tools/skill_manager_tool.py](https://github.com/NousResearch/hermes-agent/blob/main/tools/skill_manager_tool.py); README).
- **Cross-session recall:** FTS5 session search with LLM summarization as an agent tool ([tools/session_search_tool.py](https://github.com/NousResearch/hermes-agent/blob/main/tools/session_search_tool.py); README), plus optional [Honcho](https://github.com/plastic-labs/honcho) dialectic user modeling (README).
- **Dispatcher state:** cron `jobs.json` + per-job output archive; `kanban.db` holds the full task graph, claims, attempts, and each task's `session_id` — so a dispatch decision loop survives restarts by construction.

For curia's key question — "does the dispatcher hold state between runs?" — the answer is yes at every layer: the tick loops are stateless over durable SQLite/JSON stores, and the *agent* layered on top additionally accumulates memory and skills.

## Discord support

**Built-in, first-class.** Discord is a bundled platform plugin using discord.py: "Supports voice mode, slash commands, free-response channels, role-based DM auth, threads, reactions, and channel skill bindings"; `DISCORD_HOME_CHANNEL` is the default target for cron/notification delivery ([plugins/platforms/discord/plugin.yaml](https://github.com/NousResearch/hermes-agent/blob/main/plugins/platforms/discord/plugin.yaml), [adapter.py](https://github.com/NousResearch/hermes-agent/blob/main/plugins/platforms/discord/adapter.py)). Beyond relaying chat: the agent gets Discord action tools (reply/react/pin by `message_id` — [tools/discord_tool.py](https://github.com/NousResearch/hermes-agent/blob/main/tools/discord_tool.py), wiring in [gateway/run.py](https://github.com/NousResearch/hermes-agent/blob/main/gateway/run.py)); auto-created Discord threads are renamed to semantic session titles; kanban can deliver task artifacts/notifications to platform channels ([gateway/kanban_watchers.py](https://github.com/NousResearch/hermes-agent/blob/main/gateway/kanban_watchers.py) `_kanban_notifier_watcher`); HITL clarify questions render as Discord buttons ([tools/clarify_gateway.py](https://github.com/NousResearch/hermes-agent/blob/main/tools/clarify_gateway.py)). Discord threads share one session across participants by default (`thread_sessions_per_user: false`, [docs/session-lifecycle.md](https://github.com/NousResearch/hermes-agent/blob/main/docs/session-lifecycle.md) §5). Curia's "Discord bridge relays HITL questions" component would not need to be built at all.

## Voice input

**Built-in, multiple surfaces:**

- **Voice memos on chat platforms** — the gateway automatically transcribes voice messages "sent by users on Telegram, Discord, WhatsApp, Slack, and Signal"; six STT providers with **local faster-whisper as the free default** (auto-downloads ~150 MB model), plus Groq/OpenAI/Mistral/xAI/ElevenLabs ([tools/transcription_tools.py](https://github.com/NousResearch/hermes-agent/blob/main/tools/transcription_tools.py) docstring). Phone → Discord voice memo → agent is the designed path.
- **Discord voice channels** — a continuous PCM `VoiceMixer` blends an ambient "thinking" loop with TTS speech and ducking, "the smooth Grok-voice-mode feel" ([plugins/platforms/discord/voice_mixer.py](https://github.com/NousResearch/hermes-agent/blob/main/plugins/platforms/discord/voice_mixer.py)).
- **CLI push-to-talk** voice mode via sounddevice, lazily imported so headless servers don't break ([tools/voice_mode.py](https://github.com/NousResearch/hermes-agent/blob/main/tools/voice_mode.py)); TTS tools including local NeuTTS ([tools/tts_tool.py](https://github.com/NousResearch/hermes-agent/blob/main/tools/tts_tool.py), [tools/neutts_synth.py](https://github.com/NousResearch/hermes-agent/blob/main/tools/neutts_synth.py)).

## Session attach/sharing across devices

**Different model from a one-daemon/many-clients system — sharing is delegated to the chat platform.** Sessions are keyed to a platform conversation lane (`agent:main:{platform}:{chat_type}:{chat_id}…`, [docs/session-lifecycle.md](https://github.com/NousResearch/hermes-agent/blob/main/docs/session-lifecycle.md) §4); there is no protocol for two arbitrary clients to attach to one live agent session. In practice: phone-Discord and PC-Discord both see the same channel, so the Discord lane *is* the shared surface; CLI and Discord conversations are separate lanes over the same memory/state stores. Partial bridges exist: `/resume` switches a lane to another session id (`switch_session`), cross-platform deliveries are mirrored into the target session's transcript so the receiving side has context ([gateway/mirror.py](https://github.com/NousResearch/hermes-agent/blob/main/gateway/mirror.py)), and the FTS5 session search reaches across lanes. Monitoring surfaces: `hermes dashboard` (FastAPI + React SPA on port 9119) for config, API keys, "monitoring active sessions" and the kanban board ([web/README.md](https://github.com/NousResearch/hermes-agent/blob/main/web/README.md)); a TUI with a WS event sidecar ([tui_gateway/](https://github.com/NousResearch/hermes-agent/blob/main/tui_gateway)); a desktop app ([apps/desktop](https://github.com/NousResearch/hermes-agent/blob/main/apps/desktop)). For curia — user on phone via Discord, PC over Tailscale — this model is actually sufficient: both devices talk to the same Discord lanes, and the dashboard rides over Tailscale.

## Restart survival

**Designed for, in unusual depth** ([docs/session-lifecycle.md](https://github.com/NousResearch/hermes-agent/blob/main/docs/session-lifecycle.md) §7):

- Graceful shutdown writes a `.clean_shutdown` marker; on boot without it (crash), `suspend_recently_active()` marks sessions active in the last 120s as `resume_pending`.
- On startup the gateway then **synthesizes a `MessageEvent` for each `resume_pending` session "to let the agent auto-continue"** — interrupted turns are actively resumed, not just left resumable (contrast Paseo, where a dead turn waits for the next user prompt).
- A stuck-loop guard (`restart_counts.json`) suspends any session active across 3+ consecutive restarts, so a poison message can't crash-loop the gateway forever. Drain timeouts on shutdown/restart mark mid-turn sessions for resume with distinct reasons.
- Kanban workers are separately supervised: the dispatcher records worker PIDs at spawn ("a safety net for crashes, OOM kills, and Ctrl+C"), workers heartbeat, expired claims are reclaimed on the next tick, `max_runtime_seconds` bounds runaway workers, and `failure_limit` demotes repeat failures to `blocked` for human triage ([hermes_cli/kanban_db.py](https://github.com/NousResearch/hermes-agent/blob/main/hermes_cli/kanban_db.py)).
- The gateway itself runs under a service manager — systemd (Linux), launchd, Windows Scheduled Task, or s6 in containers ([hermes_cli/service_manager.py](https://github.com/NousResearch/hermes-agent/blob/main/hermes_cli/service_manager.py)) — with a restart-loop guard, shutdown watchdog/forensics, and systemd notify integration ([gateway/](https://github.com/NousResearch/hermes-agent/blob/main/gateway)).

## Self-hostability

**Excellent — a $5 VPS is the advertised target** (README: "Run it on a $5 VPS, a GPU cluster, or serverless infrastructure").

- Install: `curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash` → uv-managed Python 3.11 venv under `~/.hermes`; also Docker + docker-compose ([docker-compose.yml](https://github.com/NousResearch/hermes-agent/blob/main/docker-compose.yml)), Nix flake ([flake.nix](https://github.com/NousResearch/hermes-agent/blob/main/flake.nix)), native Windows, and Android/Termux (README).
- Models: any provider — Nous Portal, OpenRouter, OpenAI, Anthropic, "your own endpoint", local models; providers are plugins under [plugins/model-providers/](https://github.com/NousResearch/hermes-agent/blob/main/plugins/model-providers) with a registry ABC ([providers/README.md](https://github.com/NousResearch/hermes-agent/blob/main/providers/README.md)). No forced account; **Nous Portal is an optional convenience subscription** bundling model access + web search/image-gen/TTS/cloud-browser keys (README "Skip the API-key collection").
- Hosted-infra features are cleanly optional: scale-to-zero quiesce is gated behind a Nous "Labs" env stamp for Fly-style hosting ([gateway/scale_to_zero.py](https://github.com/NousResearch/hermes-agent/blob/main/gateway/scale_to_zero.py)); the Chronos managed-cron path (external one-shot arming so a hosted gateway can sleep between fires) is a plugin against Nous's account service — the default remains the local in-process 60s ticker ([docs/chronos-managed-cron-contract.md](https://github.com/NousResearch/hermes-agent/blob/main/docs/chronos-managed-cron-contract.md), [plugins/cron_providers/chronos](https://github.com/NousResearch/hermes-agent/blob/main/plugins/cron_providers/chronos)).
- Local-first defaults: STT is local faster-whisper, memory/skills/kanban/cron all on local disk. An always-on Hetzner box needs none of the serverless machinery.

## Verdict for curia

- **Role (b), cron/dispatch loop: strong fit — this ticket's role is Hermes's home turf.** Cron + webhooks + a kanban dispatcher with claims, retries, heartbeats, concurrency caps, and HITL blocking states, all persisted in SQLite, all delivered to Discord. Curia's "reads ticket frontiers, dispatches workers" loop maps 1:1 onto `ready` tasks + `dispatch_once`.
- **Role (c), bridge: also native** — Discord bot, voice memo transcription, clarify buttons; nothing to build.
- **What curia would still have to build/decide:** workers. The dispatcher spawns Hermes agents, not Claude Code/Codex. Options: (1) accept Hermes workers that drive Claude Code through the PTY terminal tool — works today, but adds an LLM intermediary per worker; (2) write a custom `spawn_fn` against `dispatch_once` (Python seam, fork-adjacent); (3) use Hermes only as dispatcher+bridge and have a thin Hermes skill/cron job shell out to curia's own worker launcher. If curia's tickets live in GitHub Issues (per current repo convention), a sync layer between GH issues and the kanban board — or webhook subscriptions on issue events — is also curia's to build.
- **Caveats:** very large, fast-moving Python monolith (a ~23k-line `gateway/run.py` mid-decomposition) — auditing or patching internals is costly; ~24k open issues means upstream fix latency is a lottery; no cross-client live session attach (acceptable given Discord is the shared surface); cron missed-fire/catch-up behavior after downtime unverified; running the full gateway (all platform adapters + local whisper + dashboard) on a small VPS is claimed but not measured here.

## Gaps / unknowns

- **Cron catch-up semantics** — whether jobs missed while the gateway is down fire on next boot or are skipped was not traced through `tick()`; worth an empirical test before relying on it for critical schedules.
- **`spawn_fn` stability** — the custom-worker seam in `dispatch_once` is internal API with no compatibility promise; sampled no docs endorsing external use.
- **Kanban behavior verified by code-reading only**, not by running a board through crash/reclaim cycles.
- **Resource footprint** of the Python gateway + local faster-whisper on a small Hetzner box — not documented, not measured.
- **Security posture under prompt injection** — there is substantial machinery ([SECURITY.md](https://github.com/NousResearch/hermes-agent/blob/main/SECURITY.md), `tools/tirith_security.py`, `tools/threat_patterns.py`, command approval, DM pairing) but it was not audited; an always-on agent with Discord ingress and a terminal tool deserves a dedicated review.
- **Issue-tracker sampling** for headless-Linux/kanban showstoppers was not done (24k open items).
