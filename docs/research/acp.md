# ACP (Agent Client Protocol) as curia's worker-facing shim seam — primary-source research

Date: 2026-07-22. Sources: spec repo `agentclientprotocol/agent-client-protocol` at commit `c3bd4ee` (2026-07-22) — cloned; docs under `docs/protocol/v1/*.mdx` (same content as https://agentclientprotocol.com); machine-readable schema `schema/v1/schema.json` + `schema/v1/schema.unstable.json`; adapter repo `agentclientprotocol/claude-agent-acp` at HEAD 2026-07-22 — cloned; npm registry metadata via `npm view`; GitHub API via `gh api`; JetBrains ACP docs and GitHub changelog fetched to verify two adopters. Naming note (verified): the repo has **moved from `zed-industries` to the `agentclientprotocol` org**; the TypeScript SDK is **`@agentclientprotocol/sdk`** (v1.3.0, 2026-07-21) — the old `@zed-industries/agent-client-protocol` (0.4.5) is deprecated with a rename notice; the Claude adapter is **`@agentclientprotocol/claude-agent-acp`** (v0.61.0, 2026-07-22) — both `@zed-industries/claude-code-acp` and `@zed-industries/claude-agent-acp` are deprecated rename shells.

**Verdict.** ACP's *agent-side* surface is small and the spec says so explicitly: the schema's `SessionCapabilities` doc-comment states "As a baseline, all Agents **MUST** support `session/new`, `session/prompt`, `session/cancel`, and `session/update`" — plus the `initialize` handshake. Everything else (load/list/resume/fork/delete/close, modes, config options, auth) is capability-gated and stubbable, and the filesystem/terminal methods are *client*-side — an agent shim implements none of them. So the protocol shim is thin; the actual weight in curia's plan is turning a PTY text stream into structured `session/update` events, which is an agent-parsing problem, not an ACP problem. The stronger finding cuts the other way: for the agents curia cares about, **the PTY may be unnecessary** — a maintained, official, standalone stdio adapter for Claude Code exists (`claude-agent-acp`, Apache-2.0, ~100 releases, several per week, maintained by ACP's lead maintainer), Codex has `codex-acp`, and Gemini CLI / Copilot CLI / many others speak ACP natively. Contract fit: `ask_human(approve-reject)` and `preview-review` map cleanly onto stable `session/request_permission` + diff tool-calls; **free-text and choice map onto `elicitation/create`, which is still unstable** (present only in `schema.unstable.json`; the SDK exposes it as `unstable_createElicitation`); `notify` is a plain `agent_message_chunk` update. Images ride both directions as base64 `ContentBlock`s (human→agent gated on `promptCapabilities.image`; agent→human ungated), and `claude-agent-acp` advertises `image: true`. Governance is jointly Zed + JetBrains, Apache-2.0 throughout, protocol version is a single negotiated integer currently at **1**, with v2 in alpha.

## Q1: Shim surface — what an ACP agent must implement

ACP is JSON-RPC over newline-delimited UTF-8 stdio; the client launches the agent as a subprocess, agent stdout carries only ACP messages, stderr is free for logging (`docs/protocol/v1/transports.mdx`; Streamable HTTP is a draft proposal only).

**Mandatory (5 things):**

- **`initialize`** — respond with chosen integer `protocolVersion` and `agentCapabilities`; capabilities omitted are treated as unsupported, so a minimal agent can advertise almost nothing (`docs/protocol/v1/initialization.mdx`; `InitializeRequest/Response` in `schema/v1/schema.json`). `authMethods` may be `[]`, making `authenticate`/`logout` moot.
- **`session/new`** — return a `sessionId`; params carry `cwd` and `mcpServers` (a shim may ignore MCP servers if it advertises no `mcpCapabilities`) (`docs/protocol/v1/session-setup.mdx`).
- **`session/prompt`** — the whole turn: accept `ContentBlock[]`, stream progress, return a `stopReason` (`end_turn`/`max_tokens`/`max_turn_requests`/`refusal`/`cancelled`) (`docs/protocol/v1/prompt-turn.mdx`; `StopReason` in schema).
- **`session/update` (outbound notification)** — the streaming channel: `agent_message_chunk`, `agent_thought_chunk`, `tool_call`, `tool_call_update`, `plan`, `available_commands_update`, etc. (`SessionUpdate` oneOf in schema). Only `agent_message_chunk` is practically needed for a minimal shim.
- **`session/cancel` (inbound notification)** — must abort and return the in-flight `session/prompt` with `stopReason: "cancelled"`, and must answer any pending `session/request_permission` with the `cancelled` outcome (`RequestPermissionOutcome` description in schema; `docs/protocol/v1/cancellation.mdx`).

The baseline quote, verbatim from `schema/v1/schema.json` → `SessionCapabilities.description`: "As a baseline, all Agents **MUST** support `session/new`, `session/prompt`, `session/cancel`, and `session/update`. Optionally, they **MAY** support other session methods… by specifying additional capabilities."

**Optional / stubbable (capability-gated, default off):** `session/load` (top-level `loadSession` cap), `session/list`, `session/resume`, `session/delete`, `session/close`, `session/set_mode`, `session/set_config_option` (all under `sessionCapabilities`), `authenticate`/`logout`, MCP passthrough, slash commands, plans, session modes. `PromptCapabilities` defaults are `image: false, audio: false, embeddedContext: false` — but curia's #12 forces `image: true`.

**Not the agent's problem at all:** `fs/read_text_file`, `fs/write_text_file`, and all five `terminal/*` methods are **client-side** (`x-side: "client"` in the schema; confirmed by enumerating every `x-method`). The agent may *call* them if the client advertised `fs`/`terminal` client capabilities; a shim whose worker does its own file/terminal I/O inside the PTY simply never calls them. `session/request_permission` and `elicitation/create` are likewise client-implemented — the agent only *sends* them.

**Estimate.** With `@agentclientprotocol/sdk` handling the wire (its `AgentSideConnection` pattern is exactly what `claude-agent-acp/src/acp-agent.ts` uses via `.onRequest(methods.agent.*)` registrations), the protocol shim is: 4 request handlers, 1 notification handler, and an event pump emitting `session/update`. That is small — days, not weeks. The genuinely heavy part of curia's "wrap a PTY-hosted session" plan is not ACP but the PTY side: parsing an interactive TUI's escape-coded output into discrete message/tool-call events and detecting "the agent is asking me something" states. ACP does not reduce that cost; it only standardizes the other side of it. (Inference, not spec: see Verdict for the alternative that removes the PTY.)

## Q2: claude-agent-acp — the ready-made Claude Code adapter

**It exists, is official, and is actively maintained.** Repo `agentclientprotocol/claude-agent-acp` (created 2025-08-27 under zed-industries, since moved; old GitHub/npm names redirect/deprecate to the new ones — verified via `gh api` and `npm view … deprecated`). Stats at 2026-07-22: 2,274 stars, 88 open issues, Apache-2.0, pushed same day. **100 GitHub releases**; latest `v0.61.0` on 2026-07-22, with v0.58–v0.61 all inside July 2026 — several releases per week. Top contributors: `benbrandt` (Ben Brandt, Zed — also ACP's lead maintainer per `MAINTAINERS.md`) with 257 commits, then `agu-z`, `ConradIrwin` (Zed) — i.e. maintained by the protocol's own core team, not a community drive-by.

- **Wraps the SDK, which wraps the CLI.** npm dependencies: `@anthropic-ai/claude-agent-sdk` 0.3.217 + `@agentclientprotocol/sdk` 1.3.0. It calls the SDK's `query()` and points it at the native Claude Code executable (`pathToClaudeCodeExecutable: … await claudeCliPath()`, `src/acp-agent.ts:5298`). So: ACP stdio ⟷ adapter ⟷ Claude Agent SDK ⟷ headless Claude Code process. No PTY, no TUI scraping.
- **Standalone outside Zed: yes.** It ships a `bin` (`claude-agent-acp` → `dist/index.js`) whose entrypoint is a plain stdio ACP server — redirects all console output to stderr, runs `runAcp()`, exits on stdin EOF (`src/index.ts`). Nothing Zed-specific in the launch path; a `--cli` flag pass-through to the raw Claude CLI also exists. It is the adapter the official ACP agents page lists for "Claude Agent" (`docs/get-started/agents.mdx`), and JetBrains' ACP client consumes external stdio agents the same way (JetBrains AI Assistant ACP docs, fetched 2026-07-22). Not empirically run in this research — inferred from source, not tested.
- **Permissions.** The SDK's `canUseTool` callback is bridged to ACP `session/request_permission` (`src/acp-agent.ts:4414` ff.); offered options include `acceptEdits` and `bypassPermissions` (bypass suppressed when running as root, `src/acp-agent.ts:1135`), and Claude Code's permission modes surface as ACP session modes.
- **Ad-hoc questions.** Notably for curia: Claude's built-in `AskUserQuestion` tool and MCP-server elicitations are both rendered as **ACP form elicitations** (`src/elicitation.ts` header comment) — via the *unstable* `elicitation/create` (`unstable_createElicitation` in `src/acp-agent.ts:1208`), degrading to permission-style flows when the client doesn't advertise elicitation support.
- **Sessions.** `session/new`/`load`/`resume`/`list` all implemented against the SDK's persisted sessions (`listSessions({dir: cwd})`, `resumeSessionAt`, `rewindFiles`); advertises `loadSession: true` plus `sessionCapabilities: {additionalDirectories, close, delete, fork, list, resume}`, `promptCapabilities: {image: true, embeddedContext: true}`, and a `_meta.steering` extension for injecting follow-ups into a running turn (`src/acp-agent.ts:1400-1450`).

## Q3: Contract fit — ask_human kinds and images

Mapping curia's escalation contract (#11) onto ACP mechanisms:

| ask_human kind | ACP mechanism | Status |
| --- | --- | --- |
| approve-reject | `session/request_permission` — agent sends `toolCall` + `options[]`; each `PermissionOption` has `optionId`, human label, and `kind` ∈ `allow_once`/`allow_always`/`reject_once`/`reject_always`; response outcome is `selected` or `cancelled` (schema `RequestPermissionRequest`, `PermissionOptionKind`, `RequestPermissionOutcome`) | **Stable v1. Exact fit.** |
| choice | `elicitation/create` form mode: restricted JSON Schema, enum/`oneOf` with per-option `title`/`description` — the RFD's worked example is literally "Which approach: conservative / balanced / aggressive?" (`docs/rfds/elicitation.mdx`) | **Unstable.** Methods exist only in `schema/v1/schema.unstable.json` (`CreateElicitationRequest`, `ElicitationFormMode`, …), absent from stable `schema.json`. Stable-only workaround: a synthetic tool call carrying the options as `PermissionOption`s — semantically abusive but wire-legal. |
| free-text | `elicitation/create` form mode with a `string` property (the RFD explicitly targets "What's the name for this function?"-style asks); url mode exists for out-of-band flows | **Unstable**, same as above. **Stable v1 has no free-text ask-the-user method** — `session/request_permission` requires a `toolCall` and closed options. |
| preview-review | `tool_call`/`tool_call_update` whose `ToolCallContent` includes the `diff` variant (or arbitrary content blocks / an embedded terminal), then `session/request_permission` on that tool call — this is exactly Zed's edit-review flow (schema `ToolCallContent`; `docs/protocol/v1/tool-calls.mdx`) | **Stable. Good fit.** |
| notify(msg) | `session/update` with `agent_message_chunk` — a JSON-RPC notification, inherently fire-and-forget | **Stable. Fit.** |

**Images (#12): yes, both directions.**

- **Human→agent:** `ContentBlock` image variant — `{type: "image", mimeType, data(base64), uri?}` — allowed in `session/prompt` when the agent advertised `promptCapabilities.image` (`docs/protocol/v1/content.mdx`; `ImageContent`, `PromptCapabilities` in schema). `claude-agent-acp` advertises `image: true`. Audio is a parallel block type behind its own capability.
- **Agent→human:** `agent_message_chunk` (and `user_message_chunk`, tool-call `content`) carry a `ContentChunk` whose `content` is any `ContentBlock` — including images; the prompt-capability gate is spec'd for the *prompt* direction only, and the schema puts no capability gate on update content (`ContentChunk`, `SessionUpdate`, `ToolCallContent` in `schema/v1/schema.json`). Content blocks are deliberately the same structure as MCP's, so MCP tool output (e.g. screenshots) forwards untransformed (`docs/protocol/v1/content.mdx`).

## Q4: Maturity

- **Versioning.** `protocolVersion` is a **single integer**, bumped only for breaking changes, negotiated in `initialize` (client sends latest supported; agent echoes it or answers with its own latest; client SHOULD disconnect on mismatch). Non-breaking features arrive as capabilities, whose introduction "is not considered a breaking change" (`docs/protocol/v1/initialization.mdx`). **Current stable protocol version: 1** (README). A v2 is in open development: `schema/v2/`, `unstable_protocol_v2` cargo feature, `schema-v2.0.0-alpha.2` release (2026-07-21). Unstable features (elicitation, mcp passthrough, providers, NES, session/fork, document sync) live in `schema.unstable.json` behind an `unstable` feature and are explicitly not covered by stability guarantees. README warns that crate/schema artifact versions do *not* imply wire compatibility — only the negotiated `protocolVersion` does.
- **Licenses.** Apache-2.0 across the protocol repo (`LICENSE`), `@agentclientprotocol/sdk`, and `claude-agent-acp` (npm metadata + GitHub API). No CLA; contributions accepted under Apache-2.0 terms (`claude-agent-acp` README).
- **Governance.** Interim model, "jointly governed by Zed and JetBrains… working toward transitioning to an independent foundation" (`docs/community/governance.mdx`). Lead maintainers: Ben Brandt (Zed) and Sergey Ignatov (JetBrains); core maintainers include Niko Matsakis; per-SDK maintainer groups (`MAINTAINERS.md`, updated 2026-06-01). Official SDKs: TypeScript, Rust, Python, Kotlin, Java (README).
- **Cadence.** Protocol repo: created 2025-06-23; 3,735 stars, 311 forks, only 26 open issues; pushed the day of this research; releases v1.4.0→v1.6.0 across 2026-07-06→07-21 (CHANGELOG + `gh api` releases). TS SDK v1.3.0 on 2026-07-21. Adapter cadence as in Q2.
- **Adopters (official lists at `docs/get-started/clients.mdx` / `agents.mdx`; two spot-verified).** Clients: Zed, **JetBrains** (verified against jetbrains.com help — ACP agents usable without an AI subscription, with a curated agent registry), Neovim (4 plugins), Emacs, VS Code extensions, Obsidian, marimo, mobile clients (Happy, etc.), and a whole *Messaging* section that includes **OpenACP (Telegram/Discord/Slack)** — curia's candidate bridge — plus ~10 sibling chat bridges. Agents: Gemini CLI, **Copilot CLI** (verified against the 2026-01-28 GitHub changelog: public preview, `copilot --acp`), Cursor, Codex CLI (via `codex-acp`), Goose, OpenCode, OpenHands, Cline, Kimi CLI, Qwen Code, Hermes Agent, and **pi via a third-party `pi-acp` adapter**. Listing ≠ verification: only JetBrains and Copilot were independently confirmed; the rest are as-listed by the ACP docs.

## Verdict for curia

- **The shim is light; the heavy part was never ACP.** An ACP agent is `initialize` + `session/new` + `session/prompt` + `session/cancel` + `session/update`, with fs/terminal living on the client side and everything else stubbable behind capabilities. The SDK removes the wire plumbing. What remains expensive in the "wrap an Orca-PTY session as an ACP agent" plan is PTY-output parsing — and that cost exists under *any* protocol choice. So shim weight is **not** an argument against the vendored-OpenACP path.
- **claude-agent-acp weighs strongly for the ACP path — and against the PTY.** For Claude Code workers, curia doesn't need to shim a PTY at all: spawn `claude-agent-acp` as a stdio subprocess and get sessions, permission requests, edit review, images, and steering from a first-party-maintained adapter (releases several times a week, maintained by ACP's lead maintainer). The same holds for Codex (`codex-acp`) and for agents with native ACP (Gemini CLI, Copilot CLI, Goose, OpenCode…). The PTY-wrapping shim shrinks to a fallback for agents with no adapter — which today is a short list.
- **One real contract gap: free-text/choice ride an unstable feature.** `ask_human(approve-reject)` and `preview-review` sit on stable v1 core; `notify` is trivial. But free-text and choice need `elicitation/create`, which is unstable (RFD stage, `schema.unstable.json` only). In practice the ecosystem is already using it — `claude-agent-acp` emits form elicitations for Claude's `AskUserQuestion` — so a vendored OpenACP fork should implement the unstable elicitation client surface and expect churn until it stabilizes (or v2 lands). Budget for that; it is the only place curia's contract outruns the stable spec.
- **Maturity is acceptable for a vendored dependency.** Apache-2.0 everywhere, two corporate stewards (Zed + JetBrains) with a stated foundation path, integer-versioned wire protocol at v1 with capability-based evolution, and a broad, partially-verified adopter list including four independent Discord/Slack/Telegram bridges to crib from if OpenACP itself disappoints.

## Gaps / unknowns

- **Nothing was executed.** `claude-agent-acp` standalone operation is inferred from its entrypoint source and its distribution model, not from a live run; same for image round-tripping through it.
- **Elicitation stabilization timeline unknown.** It is an RFD + unstable schema feature; whether it stabilizes in v1 (as a capability) or only in v2 was not determined.
- **Agent→human image rendering is client-dependent.** The schema permits image blocks in updates without a gate, but whether a given client (OpenACP, a Discord bridge) renders them is that client's business — OpenACP's handling was not inspected here (see `docs/research/openacp.md`).
- **JetBrains registry contents not enumerated** — JetBrains ships a curated ACP agent registry, but whether `claude-agent-acp` specifically is in it was not verified.
- **PTY-side cost not quantified.** The claim that PTY parsing dominates shim cost is engineering judgment, not measurement.
