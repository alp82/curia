## Role fit

OpenACP *is* a bridge ŌĆö but an opinionated one: **it insists on owning the agent processes.** Core flow (verified, `docs/gitbook/extending/building-adapters.md` and `src/core/`): platform adapter ŌåÆ `core.handleIncomingMessage()` ŌåÆ `OpenACPCore ŌåÆ Session ŌåÆ AgentInstance (ACP subprocess)` ŌåÆ agent events ŌåÆ `adapter.sendMessage() / sendPermissionRequest() / sendNotification()`. It spawns agents itself from its registry (npx/uvx/binary with SHA-256 verification) in a configured workspace. Curia's workers live in Orca, so OpenACP cannot be dropped in as a passive message pipe; the clean seam is that **any local command can run "as a generic ACP agent"** (README, "Agents and sessions"), so curia's dispatcher would present workers to OpenACP as thin ACP shims (see Integration shape). Everything else it owns ŌĆö session lifecycle, permission gating, streaming, files, speech ŌĆö is precisely the bridge machinery curia needs.

## Discord two-way HITL ŌĆö verified real

Read from the `@openacp/discord-adapter@2026.518.1` tarball's `dist/` (source repo `Open-ACP/discord-adapter` is 404; the compiled JS is readable and unminified):

- **Threads:** real. Forum channel `openacp-sessions` auto-created (Community servers get a true Forum Channel, otherwise text-channel threads), one thread per session, plus an `openacp-notifications` channel and an Assistant thread (`docs/gitbook/platform-setup/discord.md`; `dist/forums.js`, `dist/adapter.js`). Adapter capabilities: `streaming: true, richFormatting: true, threads: true, reactions: true, fileUpload: true, voice: false` (`dist/adapter.js:16-19`).
- **Allow/reject buttons:** real. `dist/permissions.js` builds a discord.js `ActionRowBuilder` of `ButtonBuilder`s from the ACP permission request's options (Ō£ģ Success / ŌØī Danger styles, `customId` `p:<nanoid8>:<optionId>`), posts it in the session thread, and fires a deep-linked notification to the notification channel. Button interaction resolves the pending request; a second/late click gets ephemeral "Permission request expired". Slash commands registered guild-scoped: `/new`, `/cancel`, `/status`, `/sessions`, `/menu`, `/handoff`, `/bypass` (auto-approve toggle), `/tts`, `/outputmode`, etc.
- **Structured forms (ACP elicitation): not rendered in Discord.** No elicitation code exists in the Discord adapter dist; docs confirm forms render natively in Telegram only, and "Adapters without a form renderer receive a visible REST fallback for non-sensitive requests" (README). Buttons yes, forms via REST.
- **External programmatic access: yes, first-class.** The in-tree API server (`src/plugins/api-server`, Fastify, default `localhost:21420`, secret-token or scoped JWT auth) exposes `POST /api/v1/sessions`, `POST /api/v1/sessions/:id/prompt`, **`POST /api/v1/sessions/:id/permission`** (resolve a pending permission from outside), **`GET/POST /api/v1/sessions/:id/elicitation[/:requestId]`** (list/answer pending form requests, owner-bound), `POST /api/v1/notify` (broadcast to all adapters' notification channels), plus an SSE adapter (`src/plugins/sse-adapter`) for live event streams (`docs/gitbook/api-reference/rest-api.md`). So an external process can both drive sessions and answer questions; the *question-raising* side, however, is reserved for the ACP agent subprocess ŌĆö an external process cannot post an arbitrary ad-hoc button-question into a thread except by being (or proxying through) the session's agent, or one-way via `/notify`.

## Voice

- **Inbound Discord voice memos ŌåÆ STT: verified.** The adapter detects voice messages, converts OGG Opus ŌåÆ WAV (`dist/adapter.js:519-524`, via the core file service using `ogg-opus-decoder`/`node-wav` ŌĆö both in core deps), and hands the audio attachment to the core speech service: native pass-through if the agent supports audio, else **local faster-whisper** (bootstrapped into an isolated Python env on first use) or **Groq** STT, transcript appended to the prompt (README "Native speech-to-text"; `src/plugins/speech`). STT is a core service, not Telegram glue.
- **TTS: verified plumbing, plugin-provided voices.** `/tts` toggle per session; agent output wrapped in `[TTS]...[/TTS]` is sent as an audio file into the thread and the marker stripped from the text draft (`dist/adapter.js:802-815`). Providers: `heavygee/openacp-openai-tts-plugin` (any OpenAI-compatible `/v1/audio/speech` endpoint ŌĆö Kokoro, Chatterbox, AllTalk, F5-TTS; npm 1.0.2, repo pushed 2026-07-21) and `Open-ACP/msedge-tts-plugin` (in the org listing; direct access 404 like the rest).

## Images both directions ŌĆö verified

- **Human ŌåÆ agent:** Discord attachments are downloaded (`downloadDiscordAttachment`, 100MB cap), MIME-classified `image`/`audio`/`file`, saved via the core file service, and attached to the prompt; empty messages with attachments get fallback text `[Photo: name]` (`dist/media.js`, `dist/adapter.js:~515-545`). So a screenshot attached to a Discord reply reaches the worker.
- **Agent ŌåÆ human:** the adapter sends files into the thread (`thread.send({ files: [{ attachment: filePath, name }] })`, `dist/adapter.js:~800`), with a polite "File too large" warning above Discord's 25MB free-tier limit. Additionally, **local automation can push a host file into one exact live session** via the acknowledged-attachment REST API (three loopback-only routes, `attachments:send` scope, success only after the adapter returns a provider message ID ŌĆö README "Connectors and automation", `src/plugins/attachment-delivery`, `src/core/attachment-delivery`). Preview screenshots into Discord: covered twice over.

## Multi-device

Supports curia's policy. Discord is inherently multi-device ŌĆö any device logged into the account sees the thread and the buttons; the permission handler settles on the **first** button click and later clicks get "expired" (`dist/permissions.js`), i.e. first-valid-answer-wins is the built-in behavior. Access is gated by `security.allowedUserIds` (empty = anyone who can message the bot; `docs/gitbook/self-hosting/security.md`). Multiple connectors (Telegram + Discord + Slack + REST/SSE) run simultaneously, but a *session* is bound to one adapter/thread (`channelId` + `threadId` in the session record), so one escalation surfaces on one platform, many devices ŌĆö not on all platforms at once.

## Restart survival ŌĆö sessions yes, pending questions no

- **Sessions: durable.** JSON file store at `<instance-root>/sessions.json` ŌĆö versioned envelope, debounced writes, atomic tmp-file rename, flush on shutdown, 30-day TTL cleanup (`src/core/sessions/session-store.ts`, verified; `docs/gitbook/features/session-persistence.md`). On restart, sessions and their thread bindings are restored; a new message in an old thread reconnects or restarts the agent with `/resume` history attach.
- **In-flight escalations: die.** `PermissionGate` is an in-memory promise with a **10-minute default timeout**, one pending per session, superseded-on-new (`src/core/sessions/permission-gate.ts`). `ElicitationGate` likewise: in-memory `Map`, 10-minute max timeout, max 8 pending (`src/core/sessions/elicitation-gate.ts`). The Discord adapter's pending-button map is also in-memory. A daemon restart (or 10 minutes of silence) kills the pending question ŌĆö the session survives, the question must be re-asked. Curia's dispatcher would need its own durable escalation record and re-ask on reconnect; OpenACP will not hold a question overnight.

## Integration shape for curia

Two viable postures:

1. **Use it as the bridge, with curia as an ACP agent.** OpenACP's worker-side interface is Zed's ACP over stdio. Curia's dispatcher ships a thin shim (built on `@agentclientprotocol/sdk`) registered as a custom agent; each escalating Orca worker surfaces through it. Then, for free: thread-per-task in Discord, streaming output, `requestPermission` ŌåÆ native allow/reject buttons (answerable from any device, first click wins), images both directions, voice memos ŌåÆ whisper transcript, TTS out, `/notify` for one-way pings, REST/SSE to observe and drive everything (`POST /sessions/:id/prompt`, `/permission`, `/elicitation`). Curia keeps its own durable escalation store and re-asks after restarts/timeouts (10-minute in-memory gate). Cost: OpenACP believes it owns the session lifecycle, so curia's dispatcher and OpenACP must agree that the shim, not OpenACP's registry, is the process boundary.
2. **Steal the design, build thin.** If the fork-dependency is unacceptable, the parts worth copying verbatim into a custom ~500-line discord.js bridge are: forum-channel/thread-per-session mapping with a notifications channel + deep links; the permission-button scheme (`p:<key>:<optionId>` customIds, first-click-settles, ephemeral "expired" replies); OGGŌåÆWAVŌåÆwhisper inbound voice; `[TTS]` block convention; the `sessions.json` versioned-envelope store with debounced atomic writes; and the `ChannelAdapter` contract itself (`docs/gitbook/extending/building-adapters.md` is effectively a spec: `handleIncomingMessage` in, `sendMessage`/`sendPermissionRequest`/`sendNotification` out).

Either way, the ACP (Zed) protocol as the worker-facing seam is the transferable idea: it is exactly an escalateŌåÆanswerŌåÆresume contract (`requestPermission`, elicitation, prompt/response) with an existing SDK and 28+ agents already speaking it.
