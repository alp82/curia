# Cline worker lane — hands-on prototype under the substrate (wayfinder #28)

Date: 2026-07-23. Method: **live execution, not source-reading** — installed the published `cline` 3.0.46 npm package (compiled linux-x64 Bun binary) on this Arch Linux box, drove `cline --acp` directly with a hand-rolled NDJSON JSON-RPC client, then registered it as a `custom` agent in the vendored OpenACP fork build from #22 (same instance root, same Discord bot) and dispatched it on a real change in an isolated git clone of curia, with the permission gate surfaced as Discord buttons. Model lane: Anthropic BYO key (`CLINE_PROVIDER=anthropic`, `CLINE_MODEL=claude-haiku-4-5`), key via env var — no Cline account, no OAuth, no TTY.

**Verdict: Cline earns a place as a curia worker lane — with the escalation shaped as "permission-gate + end-turn questions", not mid-turn elicitation.** The ACP server is real and textbook-clean: it slotted into OpenACP's custom-agent seam with a 6-line wrapper script and zero PTY parsing, spawned in ~360 ms, drove real file edits and git commands in an isolated workspace, and its `session/request_permission` rendered as Discord Allow/Reject buttons through the same bridge machinery the mock agent used in #22 — nothing Cline-specific needed anywhere. The research report's four flagged gaps (#27) all confirmed empirically, and each has a workable curia-side answer.

## Spike answers (the #27 must-answer list)

1. **Headless BYO-key boot — PASS.** `CLINE_PROVIDER=anthropic CLINE_API_KEY=… cline --acp` completes `initialize → session/new → session/prompt` with no OAuth and no TTY. Handshake ~325 ms; agent spawn under OpenACP ~360 ms; simple turn round-trip ~1.7 s on Haiku. The compiled binary runs fine on this glibc (dynamically linked ELF, ld-linux-x86-64).
   - **Provider allowlist surprise:** the shipped binary enables only a subset of the source's provider registry. `anthropic`, `openai`, `openrouter`, `ollama`, `openai-codex`, `cline` work; **`github-copilot` is "Unknown or disabled provider"** — the zero-key Copilot lane hermes rides (#21) does *not* exist for Cline. For #13 this means the Cline lane needs a real key (Anthropic/OpenRouter) or the ChatGPT-subscription OAuth.
   - **ACP mode swallows model errors.** The same bogus-provider turn that one-shot `--json` mode reports as a fatal error returns `stopReason: "end_turn"` over ACP with zero output chunks — indistinguishable from a successful empty turn. Curia's dispatcher must treat "turn ended with no message and no tool calls" as suspect (extends #21's auth-health-watchdog requirement to the ACP lane).

2. **Clarifying questions over ACP — CONFIRMED ABSENT, workable fallback.** Prompted to use any question mechanism, the model answers "NO-QUESTION-TOOL … I can ask you directly here: **What is your favorite color?**" and ends the turn. So: no `ask_question` executor, no elicitation — but **end-turn text questions + session continuity work**: a second `session/prompt` on the same session carries full conversation state (verified with a remembered codeword). Curia's bridge can relay the end-turn question to the Discord thread and feed the reply as the next prompt — the OpenACP Discord adapter already does exactly this (thread replies drive the session). HITL for a Cline worker = permission buttons (mid-turn, blocking) + question-by-end-turn (turn boundary, non-blocking). No mid-turn free-text block, unlike claude-agent-acp's unstable `elicitation/create` path (#20).

3. **Permission volume / `allow_always` — CONFIRMED, one ping per tool call.** Three tool calls in one turn → three `session/request_permission` round-trips (fired near-simultaneously when the model batches calls). Answering **`allow_always` does not stick**: the same `editor` tool re-prompted on the next turn, as the source read predicted (`ToolApprovalResult` has no "always" channel). The Discord adapter renders all three options (✅ Allow once / ✅ Allow always / ❌ Reject) — so the *human* sees an "always" button that silently means "once". Curia must implement "always" itself, filtering on the `kind` field (`read`/`edit`/`execute`/`fetch`/`other`) each request carries; auto-answering `read` alone would have removed 1 of 3 pings in the live run below.

4. **Hub daemon / crash-resume — hub suppressible; NO headless resume path.**
   - Default (`auto`) backend: `cline --acp` spawns a **detached hub daemon** (`--cline-hub-daemon`, localhost port, keyed to the workspace) that outlives the ACP process. `CLINE_SESSION_BACKEND_MODE=local` cleanly suppresses it; everything in this spike ran identically in local mode. Supervised curia deploys should set it.
   - `session/load` → **`-32601 Method not found`** despite `loadSession: true` in the capability handshake (declared-but-unimplemented, confirmed).
   - One-shot resume **`--id <sessionId>` + `--json` is broken in 3.0.46**: "JSON output mode requires a prompt argument or piped stdin" no matter how the prompt is passed (argument, stdin, either backend). So after a worker crash there is **no headless resume path at all** — crash = re-dispatch. Curia already tolerates this posture (#11/#12/#19), so it's a fit, not a blocker; but it forecloses any warm-resume ambition for the Cline lane.
   - Keys passed via `-k`/env are **persisted in plaintext** to `~/.cline/data/settings/providers.json` (per-provider, survives across runs). Deployment note: the worker user's home must be treated as key material.

5. **Images — ACP image blocks dropped (confirmed); disk-path workaround works.** A prompt carrying a proper ACP `image` content block: model says NO-IMAGE-RECEIVED. `@/path.png` mentions are *not* preprocessed over ACP (the model just sees the literal text). But **`read_files` on an image path delivers real image content to the model** — it correctly identified a solid-red PNG. So human→agent images over the Cline lane = bridge writes the attachment to the workspace and injects "see /path/to/img.png"; agent→human remains text-only (no image blocks in `session/update`, matching #27). The #12 images-both-directions requirement is met halfway by workaround; the proper fix is upstream (the CLI's own one-shot path already converts images, only the ACP translation drops them).

6. **Skills — PORT CONFIRMED.** Dropped a skill into `<ws>/.agents/skills/`; under ACP the `skills` tool advertised the full agentskills.io set — both the workspace copy and everything in `~/.agents/skills` (the complete Matt Pocock planning/dispatch/review grammar this repo uses: wayfinder, grilling, tdd, code-review, to-tickets, triage, …). Skill invocation surfaces as a `skills` tool call (kind `other`, one permission ping).

7. **Telemetry — no CLI opt-out surface found.** `cline config` only covers workflows/rules/skills/agents/plugins/hooks/mcp/tools; no telemetry command, no settings file materialized with `telemetryOptOut`. PostHog telemetry must be assumed on; a curia deploy should set the global setting by writing the settings store directly or block PostHog egress at the network layer. (Upstream issue-worthy.)

## The live golden-thread rehearsal (OpenACP + Discord)

Setup: reused #22's from-source OpenACP build + instance root unchanged; registered Cline via `agents.json` as `distribution: "custom"` pointing at a 6-line bash wrapper (exports provider/model/key env, execs `cline --acp`). That's the entire integration.

- **Headless session (adapter-less): PASS.** `api new cline <workspace>` spawned Cline through the warm-pool pipeline; a prompt ran `git log` in the workspace after OpenACP's headless auto-approve fired (the #22 caveat, reconfirmed: adapter-less sessions are not human-gated).
- **Discord-bound session: PASS.** `POST /sessions {channel: "discord"}` created a per-session thread; the real-ticket prompt (create `docs/research/assets/cline-worker-proof.md` in an isolated curia clone, then `git add`/`git commit`) drove the worker to its first `run_commands` permission, which rendered in the thread as **🔐 Permission request + ✅ Allow once / ✅ Allow always / ❌ Reject** — Cline's ACP options mapped straight onto #22's button machinery, zero new code.
- **Button click → resume: NOT closed by a human this run** — Alp was away for the whole window. What was observed instead, all live: the worker **blocked indefinitely** on the gate (one request held >25 min without dying; an earlier one failed after ~10 min when the model's parallel second request superseded it, and the model gracefully retried). The human-click→resume mechanics on this *identical* adapter+gate code path were already closed live in #22, and Cline-side answer→resume was closed in the direct-ACP harness above (permission answered → tool ran → files written) — the composition is proven in halves; only Alp's finger on this specific run is missing. Re-running it is one command against the still-registered `cline` agent.
- **The run was completed under operator bypass** (`api bypass <id> on` set *before* the prompt; bypass does **not** resolve an already-pending gate): the worker then created the proof file and committed it in ~15 s, with its final answer streamed into the Discord thread.
- **REST cannot answer an adapter-bound permission — a real bridge limitation.** The pending permission's `permissionId` (the ACP `toolCallId`) is not exposed anywhere a REST client can reach: not in `GET /sessions/:id`, not in history (pending calls absent), and the per-session SSE stream emits only heartbeats — `permission_request` events flow to adapters only. So "any device can answer" (#11) does *not* hold on vendored OpenACP: only the bound adapter's buttons can answer, and if that post fails the escalation is unanswerable (re-confirming #22's post-failure hang). **Consequence for #17/#11: curia must own the escalation record and answer surface itself; OpenACP's gate can't be driven from outside the adapter.**
- **Guild operations note:** the AI Stack guild trouble from #22 recurred in a new form — freshly created adapter channels were deleted within seconds by something guild-side (not visible to the bot; no audit-log permission). Workaround: point `forumChannelId`/`notificationChannelId` at an existing accessible channel (`#bot-test`) — the adapter then only creates *threads*, which nothing deleted. Bots can no longer create guilds via API, so the "clean throwaway guild" fix still needs a human.
- **REST/CLI quirks:** `openacp api new --channel discord` (CLI) 500s against this build while raw `POST /api/v1/sessions {channel}` works — and the failure logs nothing; the SSE-route `cancel` hangs when a gate is pending (`DELETE /sessions/:id` works). Vendoring note: prefer the REST API over the CLI wrapper, and expect to patch around the cancel path.

## Operational feel vs. the other worker lanes

| | Claude Code (PTY / claude-agent-acp) | Codex (#19 lane) | **Cline (this spike)** |
|---|---|---|---|
| Integration into bridge | official adapter (#20), stdio ACP | codex-acp, stdio ACP | native `--acp`, stdio — same seam, 6-line wrapper |
| Spawn / turn latency | heavier startup | (per #19, comparable) | ~360 ms spawn, ~1.7 s trivial turn — lightest exercised |
| Footprint | worker ~0.4 GB under Orca (#19) | similar | single compiled binary; hub daemon optional (off with one env var) |
| HITL: permission gate | v1 approve-reject | v1 approve-reject | v1 approve-reject, **every tool call, no persistent always** |
| HITL: free-text question | unstable `elicitation/create` (#20) | same protocol gap | **none mid-turn; end-turn text + re-prompt works** |
| Images in | protocol-supported (#20) | protocol-supported | **blocks dropped; disk-path workaround** |
| Model routing (#13) | Anthropic OAuth/key lanes | ChatGPT/key lanes | env-per-spawn: Anthropic key, OpenRouter, Ollama, ChatGPT-OAuth; **no Copilot** |
| Skills | native `.agents/skills` | n/a | **same `.agents/skills` layout — curia's set ports as-is** |
| Crash recovery | re-dispatch posture | re-dispatch posture | re-dispatch **only** (no resume path at all) |
| Maturity wobble | weekly official releases | official | ACP layer near-zero tests; provider allowlist churn; silent-error mode |

**Where a Cline worker fits:** the cheap, fast, skills-compatible act-mode lane — small well-specified tickets dispatched with a key-selected model, gated per tool call through the bridge, questions handled at turn boundaries. What it should *not* be handed (today): tickets needing mid-turn free-text HITL, image round-trips, or session resume. The plan/act ACP session modes (`session/set_mode`) are a genuine extra no other lane has first-class — "plan-first" tickets map onto it without process restarts.

**Feeds #17:** the custom-agent seam has now absorbed a third, unrelated vendor's agent with zero bridge changes — strong evidence the ACP-seam abstraction is the right worker-facing boundary regardless of the adopt-vs-thin outcome.
