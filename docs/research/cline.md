# Cline (cline/cline, "Cline CLI") — primary-source research, worker-lane focus

Date: 2026-07-23. Sources: repo `cline/cline` at commit `7b77622` (2026-07-23, origin/main), https://github.com/cline/cline; official docs at https://docs.cline.bot and in-repo `docs/` (same content, `docs/cli/*.mdx`); npm registry (`cline` 3.0.46); GitHub API for stats. The repo is a monorepo: `apps/vscode` (the classic VS Code extension, releases tagged `v4.x`), `apps/cli` (the standalone CLI, `@cline/cli` workspace package published to npm as **`cline`**, currently 3.0.46), `apps/cline-hub` (background daemon), and `sdk/packages/{core,llms,agents,shared,ui}` (the new TypeScript core the CLI is built on). Vendor: Cline Bot Inc. (`apps/cli/package.json` author field).

**Verdict: promising — worth a hands-on spike.** The OpenACP registry entry is **not aspirational: Cline CLI ships a genuine, native ACP stdio server**, verified in source at `apps/cli/src/acp/` — `cline --acp` speaks newline-delimited JSON-RPC over stdio using the official `@agentclientprotocol/sdk` (^0.16.1), implements `initialize` / `session/new` / `session/prompt` / `session/cancel` / `session/set_mode` / `authenticate`, streams `session/update` notifications, and routes **every tool call** through `session/request_permission` — exactly the seam curia already drives for `claude-agent-acp` and `codex-acp`, with **no PTY parsing**. Plan/act modes are first-class ACP session modes, model/provider are switchable per-session via config options or per-spawn via `CLINE_PROVIDER`/`CLINE_MODEL`/`CLINE_API_KEY` env vars, the binary is a self-contained compiled executable that runs on headless Linux (`npm i -g cline`, no VS Code, no Bun/Node runtime needed), skills live in the Claude-Code-compatible `.agents/skills` layout, and the license is Apache-2.0 from a well-resourced project (65k stars). The caveats are real but survivable: in ACP mode the **`ask_question` tool is not wired up** (question-escalation would arrive only as permission requests, not as free-form questions — a gap vs. curia's ask_human flow), the declared `image: true` prompt capability is **not actually honored** (the prompt handler drops non-text content blocks), `loadSession` is declared but not implemented, `allow_always` permission answers are not persisted (every tool call re-prompts), and ACP-mode auth is OAuth-centric (Cline account or ChatGPT subscription) with the BYO-key path going through env vars rather than the ACP `authenticate` flow. None of these kill the worker lane — curia's bridge answers permissions programmatically anyway — but the ask_question gap and the always-reprompt behavior must be confirmed and worked around in a spike.

## 1. Role fit (worker)

- **Standalone CLI, no VS Code required — confirmed.** The CLI is a separate app in the monorepo (`apps/cli/`), published to npm as `cline` ("Autonomous coding agent CLI…", npm `cline@3.0.46`, matching `apps/cli/package.json` version). Install is `npm i -g cline`; docs state headless usage explicitly: "Headless mode activates automatically with flags like `--json`, piped stdin, or redirected output" (https://docs.cline.bot/cline-cli/overview). Distribution is **compiled Bun binaries** per platform (`@cline/cli-linux-x64`, `@cline/cli-linux-arm64`, …) pulled in via `optionalDependencies` — "users need nothing pre-installed"; the CLI *cannot* run on plain Node because its TUI uses `bun:ffi` (`apps/cli/DISTRIBUTION.md`). Requires no editor, no display. Do not confuse with npm `@cline/cli` (0.0.13, "[EXPERIMENTAL] lightweight Cline CLI", bin `clite`) — a different, experimental artifact.
- **One-shot headless:** `cline --json "task"` or `git diff | cline "review changes"` — NDJSON events to stdout, auto-approve defaults to true in this mode (`docs/cli/cli-reference.mdx`; `apps/cli/src/main.ts` sets `defaultToolAutoApprove = true`). Flags: `-p/--plan`, `--auto-approve <bool>`, `-P/--provider`, `-m/--model`, `-k/--key`, `-s/--system`, `--id <session-id>` (resume), `-t/--timeout`, `--retries`, `--thinking <level>`, `--worktree` (auto-creates a detached git worktree under `~/.cline/worktrees/`), `--data-dir` (isolated state) — all defined in `apps/cli/src/commands/program.ts`.
- **Long-running interactive session: yes, via ACP** (see §2) — sessions persist across `session/prompt` calls with conversation continuity (`ensureSessionManager` keeps one `ClineCore` session manager alive per ACP session; on provider switch it replays messages into a fresh manager — `apps/cli/src/acp/acpAgent.ts`).
- **Plan/act mapping:** Cline's two modes are exposed as ACP session modes: `session/new` returns `availableModes: [plan, act]` with descriptions ("Explore the codebase and plan changes without modifying files" / "Make changes to the codebase"), default `act`; `session/set_mode` (and the equivalent `mode` config option) switches them and emits `current_mode_update` (`acpAgent.ts`). Curia mapping: dispatch = `session/new` + `session/prompt` in act mode; a "plan first" ticket label could set plan mode, collect the plan from `agent_message_chunk` updates, then flip to act — all client-side, no process restart.
- **Extras beyond the worker role** (context, not requirements): the CLI also ships a background **hub** daemon (`-z/--zen` background sessions, `apps/cline-hub`), a kanban app (`--kanban`), cron **scheduling** (`docs/cli/scheduling.mdx`), agent teams/subagents (`spawn_agent`, `teams` tools in `sdk/packages/core/src/extensions/tools/runtime.ts`), and **chat connectors** — `cline connect discord|telegram|slack|gchat|whatsapp|linear` binds incoming chat messages to agent sessions (`docs/cli/connectors.mdx`, `apps/cli/src/connectors/adapters/discord.ts`). The connectors are the *inverse* of curia's escalation bridge (chat drives the agent, rather than a mid-ticket agent escalating to chat), so they don't replace curia's Discord gate, but they signal serious headless/chat-ops investment.

## 2. ACP-native reality (the crux) — **real, verified in source**

- **Launch:** `cline --acp` — flag defined in `apps/cli/src/commands/program.ts` ("Run in Agent Client Protocol (ACP) mode for editor integration", also documented in `docs/cli/cli-reference.mdx`); `apps/cli/src/main.ts` routes it to `runAcpMode()`, which wires `AgentSideConnection` + `ndJsonStream` from **`@agentclientprotocol/sdk`** (Zed's official SDK, dependency `^0.16.1` in `apps/cli/package.json`) over `process.stdin`/`process.stdout` and blocks until the connection closes (`apps/cli/src/acp/index.ts`). Diagnostics go to stderr so stdout stays protocol-clean (`apps/cli/src/acp/auth.ts` comment: "stdin/stdout are used for the JSON-RPC transport").
- **Implemented methods** (`apps/cli/src/acp/acpAgent.ts`, `class AcpAgent implements Agent`): `initialize`, `newSession`, `prompt`, `cancel` (aborts the in-flight turn via AbortController → `sessionManager.abort`), `setSessionMode`, `unstable_setSessionModel`, `setSessionConfigOption` (provider / model / mode selects), `authenticate`, `shutdown`. Stop reasons map Cline finish reasons → ACP (`completed→end_turn`, `aborted→cancelled`, `max_iterations→max_turn_requests`).
- **Declared capabilities** (`initialize` response): `loadSession: true`, `promptCapabilities: { image: true, audio: false, embeddedContext: false }`, plus `authMethods` and `agentInfo`. Two capability claims are **overstated relative to the code**: (a) no `loadSession` method exists anywhere in `apps/cli/src/acp/` — `session/load` would fall through to the SDK's method-not-found handling; (b) `image: true`, but `prompt()` runs the request through `extractTextFromContentBlocks`, which filters to `type === "text"` blocks only — **image blocks in a prompt are silently dropped** (`acpAgent.ts`). No `elicitation` capability anywhere; grep for "elicitation" in `apps/cli/src` returns nothing.
- **`session/update` stream** (`apps/cli/src/acp/session-updates.ts`): `agent_message_chunk` (streamed text), `agent_thought_chunk` (reasoning), `tool_call` (with title, mapped `kind`, `rawInput`), `tool_call_update` (completed/failed + `rawOutput`), `current_mode_update`, `config_option_update`, `session_info_update`. No image/diff/plan content blocks are emitted — text, thoughts, and tool calls only.
- **Drop-in conclusion:** this is a textbook ACP agent built on Zed's own SDK — it should slot into OpenACP's custom-agent seam (or Zed itself) exactly like `claude-agent-acp`/`codex-acp`, zero PTY parsing. The one structural wrinkle: `ClineCore.create` in `auto` backend mode "prewarms" a **detached hub daemon** (`prewarmDetachedHubServer`, `sdk/packages/core/src/runtime/host/host.ts`) — so `cline --acp` is not a fully self-contained subprocess by default; `CLINE_SESSION_BACKEND_MODE=local` forces a local backend (same file). A spike should pin this down.

## 3. Escalation surface

- **Permissions → `session/request_permission`: yes, and aggressively.** In ACP mode the agent hard-codes `defaultToolAutoApprove: false` with `toolPolicies: { "*": { autoApprove: false } }` (`acpAgent.ts` `buildConfig`), and the runtime capability `requestToolApproval` is wired to `requestAcpToolApproval`, which translates every tool-approval request into an ACP `session/request_permission` with options `allow_once` / `allow_always` / `reject_once`, bracketed by `tool_call_update` notifications (pending → in_progress/failed) (`apps/cli/src/acp/permissions.ts`). So **every tool call (edit, command, read, fetch) escalates to the ACP client** — curia's Discord gate works unchanged mechanically, but unfiltered it would be one Discord ping per tool call. Two mitigations needed client-side: (a) curia auto-answers safe kinds (the `tool_call` carries a mapped `kind` from `mapToolKind` — read/edit/execute/fetch — to filter on); (b) note that **`allow_always` is not persisted by the agent** — `handlePermissionResponse` maps both `allow_once` and `allow_always` to a bare `{ approved: true }` (`ToolApprovalResult` has no "always" channel), so the same tool re-prompts every time; the client must implement "always" itself.
- **Free-form questions (`ask_question`) — likely NOT available over ACP.** Cline's core defines an `ask_question` tool ("Ask user a question for clarifying… 2-5 selectable options", `sdk/packages/core/src/extensions/tools/definitions.ts` `createAskQuestionTool`). But the tool is only registered when an `askQuestion` **executor** is supplied (`createDefaultTools`: `if (enableAskQuestion && executors.askQuestion …)`), and `createDefaultExecutors` provides no default for it (`sdk/packages/core/src/extensions/tools/executors/index.ts` returns readFile/search/bash/webFetch/applyPatch/editor only). The CLI's interactive modes supply one (`askQuestion: askQuestionInTerminal` in `apps/cli/src/runtime/run-agent.ts`; TUI dialog in `use-runtime-dialog-bridge.tsx`) — **but `AcpAgent` passes only `{ requestToolApproval }` and no `toolExecutors`** (`acpAgent.ts` `ensureSessionManager`), so in ACP mode the model appears to have no question tool at all. Caveat: the hub backend has a capability-contribution mechanism (`tool_executor.askQuestion`, `sdk/packages/shared/src/hub.ts`) that could theoretically inject one; a hands-on spike must confirm empirically whether an ACP session's model can ask questions. If not, curia's HITL escalation from a Cline worker is limited to permission approve/deny — clarifying questions would have to be prompted for ("ask via final message and end turn") rather than mid-turn.
- **Images:** into the agent — declared but dropped over ACP (see §2); the *CLI's own* one-shot path does support image attachments via `@/path.png` file mentions converted to data URLs (`buildUserInputMessage` in `apps/cli/src/runtime/prompt.ts`), so the capability gap is specific to the ACP translation layer. Out of the agent — no image content blocks are emitted in `session-updates.ts`. For curia's "images both directions" requirement, Cline over ACP currently fails both directions.

## 4. Model routing

- **Providers:** the `@cline/llms` package ships a large models.dev-generated registry (`sdk/packages/llms/src/providers/provider-ids.generated.ts`: anthropic, gemini, deepseek, github-copilot, bedrock, cerebras, fireworks, … dozens more) plus handwritten built-ins (`providers/builtins.ts`): **`cline`** (the managed Cline account provider, OAuth), **`openai-codex`** ("Sign in with ChatGPT Subscription" OAuth), `openrouter`, `ollama` (native local API vendor), `openai`, plus vendor implementations for anthropic/bedrock/vertex/google/mistral/openai-compatible (`providers/vendors/`). So Anthropic lane, Codex lane, OpenRouter, and local models are all expressible.
- **Per-invocation selection — yes, two ways.** (a) **Env vars at spawn** (the clean fit for curia's `model:<x>` label): `CLINE_PROVIDER`, `CLINE_MODEL`, `CLINE_API_KEY` are read by the ACP agent at `session/new`/`buildConfig` time, with env taking precedence over session state; defaults are provider `cline`, model `anthropic/claude-sonnet-4.6` (`acpAgent.ts`). When `CLINE_PROVIDER` is set, the provider config option is locked ("Cannot change provider: CLINE_PROVIDER environment variable is set"). (b) **ACP-native**: `session/new` returns `models.availableModels` + `configOptions` (provider/model/mode selects); the client can switch via `setSessionConfigOption` or `unstable_setSessionModel` mid-session. CLI one-shot equivalents: `-P/--provider`, `-m/--model`, `-k/--key` flags.
- **BYO-key vs. bundled:** the ACP `authenticate` method offers exactly two OAuth methods — `cline` (Cline account) and `openai-codex` (ChatGPT subscription) (`apps/cli/src/acp/auth.ts` `ACP_AUTH_METHODS`); OAuth in ACP mode opens a browser URL (headless box: URL is printed to stderr to open manually) and credentials persist on disk, auto-restored on later connections (`tryRestoreAuth`). **BYO API key bypasses auth entirely**: `newSession` waives authentication when `CLINE_API_KEY` is set, and `buildConfig` uses `CLINE_PROVIDER` + `CLINE_API_KEY` directly — so `CLINE_PROVIDER=anthropic CLINE_API_KEY=sk-ant-… cline --acp` should run keyed Anthropic with no account (source-verified logic; needs empirical confirmation). Docs mention `cline auth` with "Cline Provider, ClinePass, or your own provider key" (docs.cline.bot/cline-cli/overview); no free tier is claimed in the CLI docs.

## 5. Skills / config

- **Skills — Claude-Code-compatible layout: yes.** Skill search paths (`resolveSkillsConfigSearchPaths`, `sdk/packages/shared/src/storage/paths.ts`): workspace `<ws>/.clinerules/skills`, `<ws>/.cline/skills`, **`<ws>/.agents/skills`**, plus global `~/.cline/skills` and **`~/.agents/skills`** — the same `.agents/skills` directories of the agentskills.io standard that Claude Code/pi use, so **curia's existing skill set ports by directory convention**. There is also a `skills` builtin tool ("Execute a configured skill…", `runtime.ts`) whose description advertises configured skills to the model, and `cline skill` is a thin wrapper around the open `skills` npm CLI for install/update (`apps/cli/src/commands/skill.ts`).
- **Rules — AGENTS.md: yes; CLAUDE.md: no.** The instruction loader includes workspace `AGENTS.md`, global `~/.agents/AGENTS.md`, `.clinerules` (file or directory), and markdown rules files (`sdk/packages/core/src/extensions/config/user-instruction-config-loader.ts`, `paths.ts` `AGENTS_RULES_FILE_NAME = "AGENTS.md"`). No `CLAUDE.md` reference exists in the loader — curia's `CLAUDE.md → @AGENTS.md` indirection already covers this.
- **Config surface:** global state at `~/.cline` (`--data-dir` to isolate), settings dir `~/.cline/data/settings` (`--config`), MCP servers via `cline_mcp_settings.json` (loaded by `loadConfiguredMcpTools` in `runtime-builder.ts`), hooks dirs (`~/.cline/hooks`, `<ws>/.cline/hooks`, `--hooks-dir`), workflows, and per-project `.cline/` equivalents of all of the above (`paths.ts`). Note: `mcpServers` passed by an ACP client in `session/new` are stored on the session but **never used** by `buildConfig` (`acpAgent.ts`) — MCP config over ACP is a no-op today; use the settings file instead.

## 6. Maturity / license / self-host

- **License: Apache-2.0** — root `LICENSE` file is the Apache License 2.0 text; GitHub API `spdx_id: Apache-2.0`; `apps/cli/package.json` `"license": "Apache-2.0"`. No copyleft concerns.
- **Stats (GitHub API, 2026-07-23):** 64,965 stars, 6,974 forks, 1,126 open issues; repo created 2024-07-06, pushed today. Contributors: saoudrizwan (founder, ~2,255 commits), abeatrix (~851), celestial-vault, arafatkatze, sjf, BarreiroT, … — a genuine multi-maintainer company team (Cline Bot Inc.), not a bus-factor-one project. Release cadence is high and multi-artifact: extension `v4.0.10` (2026-07-20), `sdk/v0.0.65` (2026-07-19), `desktop-v0.0.3` (2026-07-22), CLI `3.0.46` on npm. Caution: the CLI+SDK stack is a **recent ground-up rewrite** (the `sdk/packages` core is distinct from the old `apps/vscode/src/core`), and the ACP layer specifically has minimal test coverage (`apps/cli/src/acp/index.test.ts` tests only a startup diagnostic) — expect churn.
- **Self-host:** compiled static binaries for linux-x64/arm64 via npm, no VS Code, no runtime deps (`DISTRIBUTION.md`); works fully offline-of-editor. One operational wrinkle: default backend mode auto-spawns a detached **hub daemon** per workspace (`prewarmDetachedHubServer`), which a supervised curia deployment should either embrace or disable (`CLINE_SESSION_BACKEND_MODE=local`).
- **Telemetry/account:** PostHog telemetry is built in (`posthog-node` dep; `getCliTelemetryService`) and **on by default** — global settings schema `telemetryOptOut: z.boolean().default(false)` with `setTelemetryOptOutGlobally` to disable (`sdk/packages/core/src/services/global-settings.ts`); there is also a feature-flags polling service. No account is required when using env-var BYO keys (§4); the `cline` provider requires the Cline account OAuth.

## What a hands-on spike must answer

1. **Does `CLINE_PROVIDER=anthropic CLINE_API_KEY=… cline --acp` actually complete `initialize → session/new → session/prompt` headless** against OpenACP's client, with no OAuth and no TTY? (Source says yes; unverified live. Also confirm the compiled binary runs on the Hetzner box's glibc.)
2. **Can the model ask a clarifying question over ACP at all?** Source analysis says `ask_question` has no executor in ACP mode and is therefore absent — confirm empirically (prompt it into a deliberately ambiguous ticket) and check whether the hub capability path resurrects it. If absent, decide whether permission-gate-only escalation is acceptable for a Cline lane.
3. **Permission volume and `allow_always`:** confirm every tool call round-trips `session/request_permission` and that `allow_always` does not stick; measure whether curia-side auto-answering by `kind` makes the Discord gate usable.
4. **Hub daemon behavior under `--acp`:** does it spawn, where does it live, does `CLINE_SESSION_BACKEND_MODE=local` cleanly suppress it, and does anything break (session persistence, `--id` resume) without it? Also test crash/kill mid-turn: what survives and does `--id` resume work from a fresh ACP connection (relevant because `loadSession` is declared-but-unimplemented)?
5. **Image behavior:** confirm image prompt blocks are dropped (and file an upstream issue — the fix looks small since the CLI's own path already handles images), and whether `@/path.png` mentions inside a text block are a workaround (the ACP prompt text passes through the same session manager).
6. **Skills port:** drop curia's `.agents/skills` set into a workspace and verify the `skills` tool advertises and executes them under ACP.
7. **Telemetry opt-out:** verify `telemetryOptOut` is settable from the CLI (`cline config`?) and honored in ACP mode.
