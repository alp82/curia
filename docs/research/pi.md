# pi (earendil-works/pi, "pi coding agent") — primary-source research

Date: 2026-07-21. Sources: repo `earendil-works/pi` at commit `65dd2e0` / release `v0.81.0` (2026-07-21), https://github.com/earendil-works/pi; docs at https://pi.dev/docs/latest; GitHub API for stats. Package: `@earendil-works/pi-coding-agent` (formerly `@mariozechner/pi-coding-agent`; repo formerly `badlogic/pi-mono`). This is the coding-agent tool by Mario Zechner ("badlogic", author of libGDX), now under Earendil Inc. — confirmed, not the ambiguously-named other "pi".

**Verdict.** pi is a **worker substrate, not an orchestrator or a bridge** — and an excellent one for that single role. It is a deliberately minimal single-session coding agent (four built-in tools: read/write/edit/bash) with a first-class Node/TypeScript SDK (`createAgentSession()`), a headless stdio RPC mode, a one-shot JSON event-stream mode, and a documented, stable, tree-structured JSONL session format on disk. That makes it the cleanest **embeddable worker** of the tools examined: curia's dispatcher would spawn/embed pi per task and stream its events. But pi is emphatically *not* the daemon or the dispatcher — by explicit design it ships **no** MCP, sub-agents, dispatch queue, cron, multi-agent orchestration, or chat bridge ("Notable Absences" on pi.dev; all delegated to extensions). It has **no Discord surface** (only a community-server link), **no voice input** of any kind built in, and **no live multi-client attach**: RPC/JSON are point-to-point subprocess stdio, not a network server, so you cannot attach another device to a running pi turn — observation from outside means owning the subprocess or tailing the session JSONL. Note the framing: Paseo (sibling eval) literally lists **pi as one of the agent providers it hosts** — which is exactly pi's place in a curia stack: the worker that an orchestrator like Paseo/Hermes drives. Strongest asset besides embeddability is **extensibility** (TypeScript extensions over 30+ lifecycle events + the Agent Skills standard, Claude Code/Codex-compatible). Maturity is strong for a young project: 74.5k stars, MIT, only 65 open issues, ~weekly releases — but a bus factor near one (badlogic ~3,441 commits; next contributor ~412) and pre-1.0 churn.

## Role fit

- **Worker (substrate): excellent, and this is pi's whole identity.** pi is a single coding agent that "gives the model four tools: read, write, edit, and bash" ([README](https://github.com/earendil-works/pi), [pi.dev](https://pi.dev)). It is built to be embedded, three ways:
  - **SDK (embed in Node):** `createAgentSession()` — "The session manages agent lifecycle, message history, model state, compaction, and event streaming"; `prompt(text)`, `subscribe(listener)`, `setModel()`, `abort()`, `dispose()`; accepts `customTools`, credential injection, custom `ResourceLoader`s. "Use it to embed pi in other applications, build custom interfaces, or integrate with automated workflows" ([docs/sdk](https://pi.dev/docs/latest/sdk)).
  - **RPC mode (`pi --mode rpc`):** headless "JSON-based protocol… for embedding the agent in other applications, IDEs, or custom interfaces." Commands are JSON lines to **stdin**, responses/events JSON lines to **stdout**. Send prompts, steer, retrieve state, observe streaming events, handle extension-UI dialogs via a request/response sub-protocol ([docs/rpc](https://pi.dev/docs/latest/rpc)).
  - **JSON event-stream mode (`pi --mode json "prompt"`):** one-shot headless invocation — pipe a prompt, get all session events as JSON lines on stdout; "Useful for integrating pi into other tools or custom UIs" ([docs/json](https://pi.dev/docs/latest/json)).
- **Dispatcher/orchestrator: no — deliberately out of scope.** pi.dev's "Notable Absences (By Design)" list is explicit: **no MCP, no sub-agents, no permission popups, no plan mode, no built-in to-dos, no background bash** — "all achievable through extensions instead." There is no ticket queue, no cron/scheduler, no worker fleet, no concurrency management. An orchestrator (curia, or Paseo/Hermes) must supply all of that and treat pi as the thing it launches.
- **Bridge: no.** No chat-platform integration of any kind (see Discord below).

## Discord support

**None (no integration).** A repo code search for `discord` returns only community links — `README.md`, `CONTRIBUTING.md`, `packages/coding-agent/README.md`, `.github/ISSUE_TEMPLATE/config.yml` (the invite `discord.com/invite/3cU7Bz4UPx` on [pi.dev](https://pi.dev)). There is no bot, no webhook, no outbound chat relay. Curia's Discord bridge would be entirely external code driving pi via the SDK/RPC/JSON interfaces — pi is the worker at the bottom of that bridge, never the bridge itself.

## Voice input

**None built in.** Repo code searches: `voice` → 4 hits, all in `CONTRIBUTING.md` / issue templates (tone/wording, not a feature); `dictation` → 0; `whisper` → 0; `transcription` → 1, and that one is only a link in `docs/skills.md` pointing to the external [pi-skills](https://github.com/badlogic/pi-skills) collection ("Web search, browser automation, Google APIs, transcription"). So the *most* pi offers is a bolt-on **transcription skill** — a helper-script/CLI packaged as an Agent Skill that would transcribe an audio *file* on demand — not live microphone/voice-memo input and not a device-to-agent voice path. For curia's "voice input from phone" requirement, pi contributes nothing native; voice would have to be solved entirely upstream (in the bridge/orchestrator) and handed to pi as text.

## Session attach/sharing across devices

**No live attach; durable resumable sessions on disk.** This is the sharpest contrast with Paseo (one-daemon/many-clients). pi is the inverse: a per-session subprocess.

- **No network server / no multi-client attach.** RPC mode "is **not** a network server… each client must spawn its own Pi process," and "You **cannot** attach to or observe an already-running Pi session from another process or device" — transport is subprocess stdio, "inherently local and point-to-point" ([docs/rpc](https://pi.dev/docs/latest/rpc)). There is no WebSocket/socket mode (repo search for a server/websocket entry: 0).
- **What you *can* do instead — read the session file.** Sessions auto-save to `~/.pi/agent/sessions/` (organized by cwd) as **JSONL trees**: every entry has `id`/`parentId`/`timestamp`, `type`-tagged (`message`, `model_change`, `compaction`, `custom`, `label`, `session_info`, …), header carries a version (current **v3**, auto-migrated on load). The format is documented for external consumption with a JS parsing example and TS types under `node_modules/@earendil-works/pi-coding-agent/dist/` ([docs/session-format](https://pi.dev/docs/latest/session-format), [docs/sessions](https://pi.dev/docs/latest/sessions)). So an external process can **tail/parse** a live session's appended lines to *observe* it, and **resume** it — but that is file-following, not a broadcast protocol, and there is no shared cursor across clients.
- **Resume:** `pi -c` (continue most recent), `pi -r` (interactive picker), `pi --session <path|id>`, `/resume`, `/name`; `/tree` navigates the branch tree and continues from any point; `/share` uploads a session as a private GitHub gist + HTML link (a static export, not live sync) ([docs/sessions](https://pi.dev/docs/latest/sessions)).
- **Consequence for curia:** to make a running pi worker observable from a phone/PC, curia's orchestrator owns the pi subprocess (SDK `subscribe()` or RPC/JSON event stream) and re-broadcasts events to whatever surface (Discord, a web UI, Paseo). pi will not do the fan-out.

## Restart survival

**Records survive; a live turn does not (same shape as Paseo).** All conversation state is durable JSONL on disk (`~/.pi/agent/sessions/…`), and extensions can persist their own state across restarts via `pi.appendEntry()` / `custom` entries ([docs/extensions](https://pi.dev/docs/latest/extensions), [docs/session-format](https://pi.dev/docs/latest/session-format)). But pi is a foreground process, not a supervised daemon: there is no built-in service manager, auto-relaunch, or mid-turn resume. If the pi process (or its parent) dies mid-turn, the in-flight turn is lost; history up to the last flushed entry survives and the session resumes on the next `pi -c`/`--session` invocation. Supervision, restart, and turn-retry are curia's responsibility (systemd around the orchestrator that spawns pi), not pi's.

## Self-hostability

**Excellent — it is a plain npm CLI/library with no service dependency.** Install `@earendil-works/pi-coding-agent` via npm; runs anywhere Node runs. **15+ model providers** (Anthropic, OpenAI, Google, Azure, Bedrock, Mistral, Groq, HuggingFace, …) with mid-session model/provider switching and **custom providers/models** ([docs/providers](https://pi.dev/docs/latest/providers), [docs/custom-provider](https://pi.dev/docs/latest/custom-provider), [pi.dev](https://pi.dev)). Explicit **Containerization** docs ([docs/containerization](https://pi.dev/docs/latest/containerization)) and platform guides for Windows, **Termux on Android**, tmux ([docs/termux](https://pi.dev/docs/latest/termux), [docs/tmux](https://pi.dev/docs/latest/tmux)). No forced account, no telemetry surface, no daemon to keep alive — provider API keys are the only external dependency. On a Hetzner box it is a dependency of curia's orchestrator, not a separately-managed service. Footprint is negligible next to a daemon-plus-local-models design (pi ships no local ASR/TTS models because it has no voice).

## Extensibility (special focus — pi's core strength)

- **TypeScript extensions.** Default-export factory `(pi: ExtensionAPI) => {…}`, compiled at runtime via jiti (no build step). Extensions can: register LLM-callable **custom tools** (`pi.registerTool()`), hook **30+ lifecycle events** (e.g. `tool_call` to block/inject/modify), add **slash commands** and keyboard shortcuts, drive **UI dialogs/confirmations**, build **TUI components**, and **persist state** across restarts (`pi.appendEntry()`). Auto-discovered from `~/.pi/agent/extensions/*.ts` (global) and `.pi/extensions/*.ts` (project, after a trust check), or `pi -e ./path.ts` ([docs/extensions](https://pi.dev/docs/latest/extensions)). Because pi ships almost nothing by default, extensions are the intended way to add MCP, sub-agents, permission gates, todos, etc.
- **Skills (Agent Skills standard).** "Self-contained capability packages that the agent loads on-demand." pi implements the [agentskills.io](https://agentskills.io/specification) spec and is **compatible with Claude Code and OpenAI Codex skills** — you can point pi at their skill directories. Installed under `~/.pi/agent/skills/`, `~/.agents/skills/`, `.pi/skills/`, `.agents/skills/`, or `--skill <path>`; invoked as `/skill:<name>` commands whose descriptions are injected into the prompt ([docs/skills](https://pi.dev/docs/latest/skills)). This means skills curia already writes for Claude Code/Codex carry over to pi workers unchanged.
- **Packages, prompt templates, themes, custom models.** A `pi packages` registry ([pi.dev/packages](https://pi.dev/packages)) plus prompt-template and theme customization ([docs/prompt-templates](https://pi.dev/docs/latest/prompt-templates), [docs/themes](https://pi.dev/docs/latest/themes)).

## Verdict for curia

- **Role: worker substrate — strong, clean fit.** If curia wants a Claude-Code/Codex-class coding agent it *embeds and controls* rather than shells out to blindly, pi is arguably the best-shaped candidate: SDK for in-process embedding, RPC/JSON for subprocess control, custom-tool injection, a stable observable session file, MIT license, 15+ providers, and Claude-Code-compatible skills so curia's existing skills port over. This is the layer Paseo/Hermes would *drive*.
- **Role: orchestrator/dispatcher — no.** pi has, by design, none of curia's dispatch machinery (no queue, cron, multi-agent, concurrency caps, HITL states). It is the opposite end of the stack from Hermes's kanban dispatcher.
- **Role: bridge — no.** No Discord, no chat integration, no voice. Curia builds the entire bridge/voice/attach layer above pi.
- **What curia would build around pi:** (1) a supervisor that spawns one pi subprocess (or SDK session) per task, restarts it, and enforces concurrency; (2) an event fan-out that turns pi's stdio/SDK event stream into a device-observable surface (since pi offers no multi-client attach); (3) all Discord + voice + HITL, feeding pi text only.
- **Natural pairing:** since Paseo already lists pi as a hosted provider, one concrete stack is **Paseo/Hermes as daemon+dispatcher+bridge, pi as a worker provider** — pi supplies the disciplined, MIT-licensed, extensible agent core; the orchestrator supplies everything pi deliberately omits.

## Gaps / unknowns

- **Live-observation ergonomics unverified.** The "tail the session JSONL to observe a running worker" path is inferred from the documented append-only v3 format; not tested for latency/partial-line safety while pi is mid-write. The clean path is owning pi via SDK `subscribe()` / RPC events, not file-watching.
- **Pre-1.0 format/API churn.** Session format is at v3 with auto-migration and the SDK/RPC surfaces may move before 1.0; anything curia builds against internals should expect maintenance.
- **Restart/mid-turn behavior confirmed by docs only**, not by an empirical kill-mid-turn test; no built-in retry.
- **Bus factor ~1** despite Earendil backing; low open-issue count is encouraging but not a substitute for a broader maintainer base.
- **Transcription skill not inspected.** The pi-skills "transcription" entry was noted from a docs link only; whether it is file-based (likely) vs. any streaming path was not confirmed — but it is external to core pi regardless.
